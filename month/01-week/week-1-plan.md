# Неделя 1: Стена. Базовые структуры и асимптотика

**Принцип:** Каждое задание ведёт к навыку, который будешь использовать. Никакого кода ради галочки. Боль — от того, что делаешь без подсказок и разбираешь ошибки сам; польза — от того, что потом этим пользуешься.

**Книга:** «Introduction to Algorithms» (CLRS), 3-е издание. Читай только указанные разделы.

**Ориентир:** ~3 ч в день. Ниже у каждого блока есть «Зачем» — чтобы было ясно, на что ты тратишь усилие.

---

## Зачем эта неделя

- **Теория (CLRS):** Асимптотика O/Θ — язык, на котором говорят про алгоритмы. Без него не понять, почему один код масштабируется, другой нет. Массив, список, стек, очередь в книге — чтобы знать контракт (что за операции, какая цена) до того, как писать код.
- **Массив в C:** В C нет «массива с ростом». Либо фиксированный буфер, либо ты сам делаешь malloc/realloc и копирование. Это база для всего дальнейшего в системном коде. В JS/PHP нативный массив уже это умеет — дублировать бессмысленно.
- **Список, стек, очередь в JS/PHP:** В высокоуровневых языках нет «связного списка» и явного стека/очереди как основных типов. Реализовать их самому — значит понять ссылки, интерфейс (LIFO/FIFO) и когда что применять (скобки → стек, BFS → очередь). Это пригодится в задачах и на собесах.
- **Kata:** Закрепление на задачах того, что прочитал и написал. Часть задач — с явным использованием своей структуры (например Valid Braces своим стеком).
- **Чужой код:** Привычка читать, держать в голове, воспроизводить без копипаста. Это то, чем ты будешь заниматься в реальных репо и при ревью.

## День 1: Асимптотика и массив в C

**Зачем:** Без O(n) и «удвоения буфера» не разговаривать о производительности. В C массив с ростом — первый серьёзный кусок, на котором ломаются все; после него остальные структуры в C проще.

**Теория**

- [x] CLRS: глава 1 — разделы 1.1, 1.2; глава 2 — разделы 2.1, 2.2. Цель: что такое алгоритм, что время работы зависит от размера входа n, как разбирают алгоритм по шагам (на примере сортировки). О и Θ появятся в главе 3 (день 5).

**Практика (только C)**

- [x] Реализовать **динамический массив в C**: `malloc`/`realloc`, при переполнении — увеличить буфер (удвоить), скопировать элементы. Интерфейс: создать, push, pop, get по индексу, set по индексу, размер, освободить память. Без готовых векторов — только стандартная библиотека C.

**Codewars**

- [x] [Sum of positive](https://www.codewars.com/kata/sum-of-positive) (6 kyu).
- [x] [Reversed sequence](https://www.codewars.com/kata/reversed-sequence) (6 kyu).  
       Обе или одну — по времени.

**Чужой код**

- [x] **C (после своей реализации массива):** [tidwall/buf.c — buf.c](https://github.com/tidwall/buf.c/blob/master/buf.c) — первые 40–60 строк или целиком. Прочитать → закрыть → переписать логику роста буфера сам → сравнить.
- [x] **Или JS:** [Ramda filter.js](https://github.com/ramda/ramda/blob/master/source/filter.js). Прочитать → закрыть → переписать логику сам → сравнить.

**Лог:** что сделал, что сломал, что понял.

## День 2: Связный список

**Зачем:** Список даёт O(1) вставку/удаление при известном узле и не требует непрерывной памяти. В JS/PHP нет его «из коробки» — реализация учит ссылкам и обходу. Основа для стеков/очередей на списках и для графов позже.

**Теория**

- [ ] CLRS, глава 10: раздел про **связный список** (singly linked). Insert/Delete, обход O(n).

**Практика (JS или PHP — один язык)**

- [ ] Реализовать **singly linked list**: класс Node (value, next), класс списка с методами append, prepend, delete(value) — первый найденный, find(value), length. Без готовых коллекций и SplDoublyLinkedList.

**Codewars**

- [ ] [Array.diff](https://www.codewars.com/kata/array-dot-diff) (6 kyu).
- [ ] [Remove First and Last Character](https://www.codewars.com/kata/remove-first-and-last-character) (8 kyu).  
       Обе или одну.

**Чужой код**

- [ ] **C:** [Gist: связный список на C](https://gist.github.com/ArnonEilat/4470948). Прочитать → закрыть → переписать ключевые функции (добавление, удаление, обход) сам → сравнить.
- [ ] **Или JS:** [Ramda find.js](https://github.com/ramda/ramda/blob/master/source/find.js) или [take.js](https://github.com/ramda/ramda/blob/master/source/take.js). Прочитать → закрыть → переписать → сравнить.

**Лог.**

## День 3: Стек

**Зачем:** Стек — LIFO, вызовы функций, парсеры, скобки, откат. Реализация фиксирует интерфейс (push, pop, peek, size) и сразу применяется в задаче на скобки.

**Теория**

- [ ] CLRS, глава 10: раздел **стек** (Stack). Push, Pop, где используется.

**Практика (JS или PHP — другой язык относительно дня 2)**

- [ ] Реализовать **стек**: push, pop, peek (или top), size. Внутри — свой список из дня 2 или массив; снаружи только эти методы.

**Codewars**

- [ ] [Valid Braces](https://www.codewars.com/kata/valid-braces) (6 kyu) — решить **своим классом стека**, не встроенным массивом.

**Чужой код**

- [ ] [Ramda pipe.js](https://github.com/ramda/ramda/blob/master/source/pipe.js) — композиция «по цепочке», по смыслу близко к стеку вызовов. Прочитать → закрыть → переписать логику сам → сравнить.
- [ ] **Или** [Ramda drop.js](https://github.com/ramda/ramda/blob/master/source/drop.js). Прочитать → закрыть → переписать → сравнить.

**Лог.**

## День 4: Очередь

**Зачем:** Очередь — FIFO, BFS, задачи, очереди сообщений. Та же идея: явный контракт (enqueue, dequeue, front, size) и понимание, когда применять.

**Теория**

- [ ] CLRS, глава 10: раздел **очередь** (Queue). Реализация на массиве (кольцо) или на списке.

**Практика (JS или PHP)**

- [ ] Реализовать **очередь**: enqueue, dequeue, front (или peek), size. На списке из дня 2 (с хвостом) или на массиве с head/tail — как удобнее.

**Codewars**
