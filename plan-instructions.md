# Инструкции по составлению и правке планов (для ИИ / нейросети)

Этот файл описывает правила и нюансы, которые нужно учитывать при создании или редактировании планов обучения в этом репозитории (в т.ч. `plan.md`, планы недель в `месяц/N неделя/`). Все пункты ниже — пожелания пользователя, проверенные в диалоге.

---

## 1. Формат плана: конкретные задачи, не время

- **Не писать:** «1–2 часа на задачи», «40 минут на Codewars», «30% теория / 70% практика» как главный ориентир.
- **Писать:** чёткий список дел с чекбоксами: «Прочитай это», «Сделай это», «Реши эти задачи». Время можно указать один раз в начале (например «ориентир ~3 ч в день») и не дублировать по блокам.
- Пользователь хочет видеть **что делать**, а не **сколько минут тратить**.

---

## 2. Языки: JS, PHP, C (и TS по желанию)

- В плане учитывать **PHP** наравне с JS, не только «JS и C».
- **Массив (динамический)** — реализовывать **только в C**. В JS/PHP нативный массив уже динамический; дублировать его реализацию бессмысленно и пользователь это явно просил убрать.
- В **JS/PHP** фокус на том, чего нет «из коробки»: **связный список, стек, очередь**. Так есть польза: интерфейс, когда что применять, подготовка к задачам и собесам.

---

## 3. Польза каждого задания

- Никакого «программирования ради галочки». У каждого блока должен быть явный **«Зачем»**: зачем эта теория, эта структура, эта задача.
- Если задание не даёт навыка, который потом будет использоваться (в задачах, в C, в ревью кода) — его не включать или переформулировать.
- План воспринимается как **«стена»**: преодоление через боль и усилие, чтобы стать сильным специалистом. Задания должны быть сложными, но осмысленными.

---

## 4. Книги и теория

- Указывать **полное название книги** и, при необходимости, авторов. Например: «Introduction to Algorithms» (Cormen, Leiserson, Rivest, Stein), 3-е издание, сокращённо CLRS.
- Для теории давать **конкретные разделы/главы** (например «гл. 1, разделы 1.1, 1.2»), а не только «прочитай главу 2».

---

## 5. Задачи (Codewars, LeetCode, Codeforces)

- Давать **конкретные задачи с названиями и ссылками**, а не «2–3 kata на выбор» без списка.
- Указывать уровень (6 kyu, 5 kyu и т.д.) и кратко — суть задачи (например «проверка скобок», «сумма положительных»).
- Если задача должна решаться **своей реализацией** (например Valid Braces своим стеком) — явно это писать.

---

## 6. Чужой код (чтение и переписывание)

- **Обязательно давать ссылки** на реальные репозитории и файлы. Не писать «40–60 строк из любого модуля» без вариантов.
- Ссылки **проверять** (репо и файл существуют, URL актуален). При сомнении указывать альтернативы.
- Для каждого дня (или недели) предлагать **конкретные варианты** под тему: JS (например Ramda), C (например buf.c, gist), PHP (например php-array-helpers). Объём: один маленький файл или 40–60 строк из файла.
- Формат задания: «Прочитать → закрыть → переписать логику сам → сравнить».

---

## 7. Главный план (plan.md)

- **Дни и месяцы** — это **последовательность этапов**, а не календарь. Не привязывать к датам; темп задаёт пользователь.
- **Не указывать нормы по времени** (часы в день, минуты на блок). Только шаги: что прочитать, что реализовать, какие задачи, какой чужой код.
- Оформление в стиле недельного плана: блоки **Теория**, **Практика**, **Задачи**, **Чужой код**, **Итог** по этапам; при необходимости чекбоксы.
- Уже встроены в план: **Docker**, **WebSockets**, **RabbitMQ/Kafka** — при добавлении новых недель/месяцев учитывать их в нужных местах (см. текущий plan.md).

---

## 8. Структура репозитория

- Главный план: `plan.md` в корне.
- Планы по неделям: `месяц / N неделя / week-N-plan.md`.
- Лог: `месяц / N неделя / log.md` (пользователь ведёт его сам; в плане можно напоминать «записать в log.md»).
- Этот файл с инструкциями: `plan-instructions.md` в корне — использовать при генерации или правке любых планов.

---

## 9. Рефакторинг и лог

- В плане заложено: **рефакторинг каждые 7–10 дней** (переписать старый код короче/проще).
- В конце недели — **лог недели**: что сделал, сколько задач, какие структуры, главный инсайт, что было тяжело. Ссылка на `log.md` и при необходимости на `plan.md` (следующий блок этапов).

---

## 10. Правила пользователя (не менять в плане)

- Без подсказок, StackOverflow и готового кода — только документация, книги, свои решения.
- Застрял — разбирать ошибку вручную (gdb/valgrind для C, console.log для JS и т.п.).
- Чтение чужого кода: прочитать → закрыть → переписать сам → сравнить.
- Языки: чередовать JS/TS/PHP с C.
- Веди лог (что сделал, что сломал, что понял).

---

## 11. Как проверять себя (самопроверка без подсказок)

Пользователь делает задания самостоятельно; нужны способы убедиться, что реализация верна, без поиска готовых решений.

- **Свои структуры (массив, список, стек, очередь):** писать простые тесты самому: вызвать операции, проверить результат (например: push 1,2,3 → pop даёт 3; size после трёх push равен 3; enqueue 1,2 → dequeue даёт 1). Не глядеть чужой код — только свои инварианты (что должно быть истиной после каждой операции).
- **Codewars / LeetCode / Codeforces:** встроенные тесты задачи уже проверяют. Если задача сдаётся — решение корректно для этих кейсов. Использование своей структуры в задаче (например Valid Braces своим стеком) одновременно проверяет и задачу, и структуру.
- **C-код:** запускать под **valgrind** (или аналог) — утечки и выход за границы покажет. Это не «подсказка», а проверка корректности памяти.
- **Стресс:** для массива/стека/очереди — прогнать много операций (например 10k–100k), потом проверить размер и несколько значений. Если не падает и инварианты выполняются — реализация скорее верна.
- В планах недель при необходимости добавлять пункт: «Написать 3–5 простых проверок (тестов) для своей структуры; прогнать под valgrind (C)».

При составлении нового плана (новая неделя, новый месяц или правка существующего) — следовать этим пунктам, чтобы план не приходилось многократно исправлять и он оставался разумным и полезным.
